## UIC 是什么

**UIC（User Interface Compiler）** 是 Qt 工具链中的代码生成工具。`UIC` 读取 `.ui` 文件，输出对应的 C++ 头文件。

`.ui` 文件本质是 XML 文档，由 Qt Designer 生成或手动编写，描述窗口的控件层次、属性和布局：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<ui version="4.0">
 <class>MainWindow</class>
 <widget class="QMainWindow" name="MainWindow">
  <widget class="QWidget" name="centralwidget">
   <widget class="QPushButton" name="pushButton">
    <property name="text"><string>Click Me</string></property>
   </widget>
  </widget>
 </widget>
</ui>
```

UIC 在构建流程中的位置：

```
mainwindow.ui  →  uic  →  ui_mainwindow.h  →  编译器  →  目标文件
```

UIC 不参与运行时，仅在预处理阶段之前执行一次。



## 构建流程配置

CMake 中 UIC 配置如下。

```cmake
cmake_minimum_required(VERSION 3.16)
project(MyApp)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_AUTOUIC ON)
set(CMAKE_AUTOMOC ON)

find_package(Qt6 REQUIRED COMPONENTS Core Gui Widgets)

add_executable(MyApp
    main.cpp
    mainwindow.cpp
    mainwindow.h
    mainwindow.ui
)

target_link_libraries(MyApp PRIVATE Qt6::Core Qt6::Gui Qt6::Widgets)
```

`CMAKE_AUTOUIC ON` 使 CMake 在检测到 `.ui` 文件时自动调用 UIC，生成的头文件放入构建目录。



## 生成的 `ui_xxx.h` 结构

对应前面 XML 示例，UIC 生成的 `ui_mainwindow.h` 核心结构如下：

```cpp
// ui_mainwindow.h（由 UIC 自动生成，勿手动修改）

class Ui_MainWindow {
public:
    QWidget     *centralwidget;  // <widget name="centralwidget">
    QPushButton *pushButton;     // <widget name="pushButton">

    void setupUi(QMainWindow *MainWindow) {
        centralwidget = new QWidget(MainWindow);
        pushButton    = new QPushButton(centralwidget);
        pushButton->setText("Click Me");  // <property name="text">
    }

    void retranslateUi(QMainWindow *MainWindow) { ... }
};

namespace Ui {
    class MainWindow : public Ui_MainWindow {};
}
```

| 成员 | 说明 |
|---|---|
| 控件指针（如 `pushButton`） | 对应 `.ui` 中每个命名控件 |
| `setupUi(QWidget*)` | 创建所有控件，设置属性，挂入对象树 |
| `retranslateUi(QWidget*)` | 集中处理所有需翻译的字符串，供 `QTranslator` 调用 |

`namespace Ui` 中的 `class MainWindow` 继承自 `Ui_MainWindow`，是外部代码引用的实际类型。



## 三文件协作机制

| 文件 | 职责 |
|---|---|
| `mainwindow.ui` | 声明 UI 结构（XML），由 Designer 维护 |
| `ui_mainwindow.h` | UIC 生成的 C++ 代码，包含控件创建逻辑 |
| `mainwindow.h` | 声明业务类 `MainWindow`，持有 `Ui::MainWindow` 指针 |
| `mainwindow.cpp` | 实现业务逻辑，调用 `ui->setupUi(this)` 初始化界面 |

**mainwindow.h**

```cpp
#include <QMainWindow>

QT_BEGIN_NAMESPACE
// 前向声明：告知编译器 Ui::MainWindow 存在，避免包含生成头文件
namespace Ui { class MainWindow; }
QT_END_NAMESPACE

class MainWindow : public QMainWindow {
    Q_OBJECT
public:
    explicit MainWindow(QWidget *parent = nullptr);
    ~MainWindow();
private slots:
    void on_pushButton_clicked();
private:
    Ui::MainWindow *ui;  // 持有 Ui_MainWindow 实例的指针，描述"界面长什么样"
};
```

**mainwindow.cpp**

```cpp
#include "mainwindow.h"
#include "ui_mainwindow.h"  // 引入 UIC 生成的头文件，Ui::MainWindow 定义在此

MainWindow::MainWindow(QWidget *parent)
    : QMainWindow(parent)
    , ui(new Ui::MainWindow)   // 步骤1：创建 Ui_MainWindow 实例
{
    ui->setupUi(this);         // 步骤2：以 this（MainWindow）为父对象创建所有控件
                               //        控件挂入对象树，生命周期由 MainWindow 接管
}

MainWindow::~MainWindow() {
    delete ui;                 // 步骤3：释放 Ui_MainWindow 结构体本身
                               //        控件已在对象树中，由 Qt 自动销毁，无需手动 delete
}

void MainWindow::on_pushButton_clicked() {
    // 步骤4：通过 ui 指针访问控件，实现业务逻辑
    ui->pushButton->setText("Clicked");
}
```

### 为什么分成两个对象

`Ui_MainWindow` 和 `MainWindow` 是两个职责不同的对象：

| 对象 | 职责 |
|---|---|
| `Ui_MainWindow`（`ui` 指针所指） | 描述界面结构，持有控件指针，不拥有控件生命周期 |
| `MainWindow` | 实际窗口，作为所有控件的父对象，负责业务逻辑 |

`setupUi(this)` 是建立两者关联的关键：它将 `MainWindow` 作为根父对象传入，使所有控件挂入以 `MainWindow` 为根的对象树。此后 `Ui_MainWindow` 只作为访问控件的入口，控件的所有权归 `MainWindow`。

这种分离使界面结构（`.ui` 文件）与业务逻辑（`.cpp` 文件）互不干扰：修改界面只需编辑 `.ui` 文件并重新生成 `ui_mainwindow.h`，业务代码无需改动。



## `ui_xxx.h` 为何没有 `Q_OBJECT`

`Ui_MainWindow` 不包含 `Q_OBJECT` 宏，原因在于它不需要元对象系统的任何功能：

- 不定义信号或槽
- 不需要动态属性（`setProperty` / `property`）
- 不需要运行时类型识别（`qobject_cast`）

`Q_OBJECT` 会触发 MOC 为该类生成 `moc_xxx.cpp`。若 `Ui_MainWindow` 包含此宏，每次 `.ui` 文件变更都会触发 MOC 重新生成，增加构建开销，且无任何收益。

信号槽由业务类 `MainWindow` 承担，`Ui_MainWindow` 仅作为其成员，提供控件访问入口。

| 类 | `Q_OBJECT` | 职责 |
|---|---|---|
| `Ui_MainWindow` | 无 | 控件创建与初始化 |
| `MainWindow` | 有 | 信号槽、业务逻辑、元对象系统 |



## 控件的内存管理

### `ui` 指针需手动 `delete`

```cpp
MainWindow::~MainWindow()
{
    delete ui;  // ui 指向的是 Ui_MainWindow 实例
}
```

`delete ui` 释放的是 `Ui_MainWindow` 这个结构体本身，而非它持有的那些控件。`Ui_MainWindow` 析构时，仅释放其成员变量（`QPushButton*`、`QLabel*` 等指针变量）占用的空间，不会 `delete` 它们所指向的控件对象。

控件的销毁由 Qt 对象树机制完成：`setupUi(this)` 将 `MainWindow` 设为所有控件的根父对象，`MainWindow` 析构时，Qt 自动递归销毁整棵对象树。

### 对象树结构

```
MainWindow (QMainWindow)          ← 控件所有者，析构时触发递归销毁
├── centralwidget (QWidget)
│   └── pushButton (QPushButton)
```

`Ui_MainWindow` 不继承 `QObject`，不在对象树中，因此不会被自动释放，必须手动 `delete`。



## 替代方案对比

### 纯代码模式

不使用 `.ui` 文件，直接在 C++ 中创建控件：

```cpp
MainWindow::MainWindow(QWidget *parent) : QMainWindow(parent) {
    auto *central = new QWidget(this);
    auto *button  = new QPushButton("Click Me", central);
    auto *layout  = new QVBoxLayout(central);
    layout->addWidget(button);
    setCentralWidget(central);
    connect(button, &QPushButton::clicked, this, &MainWindow::onButtonClicked);
}
```

### `QUiLoader` 动态加载模式

在运行时从 `.ui` 文件加载界面，无需 UIC 预编译：

```cpp
#include <QUiLoader>
#include <QFile>

QFile file("mainwindow.ui");
file.open(QFile::ReadOnly);
QUiLoader loader;
QWidget *widget = loader.load(&file, this);
file.close();

// 控件只能通过名称查找
QPushButton *btn = widget->findChild<QPushButton*>("pushButton");
```

需在 `CMakeLists.txt` 中添加 `Qt6::UiTools`：

```cmake
find_package(Qt6 REQUIRED COMPONENTS UiTools)
target_link_libraries(MyApp PRIVATE Qt6::UiTools)
```

### 三种方式对比

| 维度 | 纯代码模式 | UIC 编译模式 | QUiLoader 动态模式 |
|---|---|---|---|
| 编译期类型检查 | 有 | 有 | 无 |
| 运行时灵活性 | 低 | 低 | 高（可热更新） |
| 类型安全 | 高 | 高 | 低（需 `findChild`）|
| 额外构建依赖 | 无 | UIC | `uitools` 模块 |
| Qt Designer 支持 | 无 | 完整 | 完整 |
| 适用场景 | 动态/程序化 UI | 常规应用开发 | 插件、皮肤、动态布局 |



## 总结

UIC 的核心作用是将 `.ui` 文件在构建期转换为 C++ 代码，使界面定义与业务逻辑在文件层面分离。

这套机制的关键点：

- **`Ui_MainWindow`** 只描述界面结构，不拥有控件生命周期
- **`setupUi(this)`** 将 `MainWindow` 设为所有控件的父对象，控件生命周期由对象树接管
- **`delete ui`** 释放的是 `Ui_MainWindow` 结构体本身，不影响控件
- **`ui_xxx.h`** 无需 `Q_OBJECT`，信号槽由业务类承担

常规 Qt Widgets 应用选择 UIC 编译模式：编译期类型安全，Designer 完整支持，运行时无额外开销。
