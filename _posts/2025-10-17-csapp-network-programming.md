## 一、网络编程基础

### 1、客户端-服务器模型

网络应用基于**客户端-服务器模型**构建。客户端和服务器均指进程，而非物理主机。一台主机可同时运行多个客户端和服务器进程，客户端与服务器的事务可在同一主机或不同主机上执行。

网络应用的基本操作是事务，包含四个步骤：
1. 客户端发送请求
2. 服务器处理请求
3. 服务器发送响应
4. 客户端处理响应

### 2、网络作为 I/O 设备

从主机角度看，网络是一种 I/O 设备。操作网络使用标准的文件 I/O 接口：`open`、`close`、`read`、`write`。这种抽象使网络编程与文件操作保持一致的编程模型。

### 3、网络设备与协议分层

以太网和 Wi-Fi 均为局域网技术，工作在 OSI 模型的 L2 数据链路层。数据链路层负责帧封装、MAC 地址寻址、冲突避免和错误检测。

网络接口层包含 L1 物理层和 L2 数据链路层，其上是网络层（IP 层），用于定位主机。

#### 网络设备的层次划分

**集线器（Hub）**：L1 物理层设备，仅执行信号接收和放大。所有连接设备共享同一冲突域，效率低下。

**网桥（Bridge）**：L2 数据链路层设备，能够解析数据帧并读取 MAC 地址。通过查询 MAC 地址表执行转发，实现冲突域隔离。

**交换机（Switch）**：多端口网桥，每个端口构成独立的冲突域。交换机接收数据帧后，根据目标 MAC 地址查表并转发至对应端口，是局域网的主流设备。

**路由器（Router）**：L3 网络层设备，解析 IP 数据包并查询路由表执行转发。路由器的每个物理接口配置独立的 IP 地址，用于网段隔离。该 IP 地址作为所连接网段的网关（Gateway），负责不同网络间的数据转发。

工作在高层的设备能够解析低层协议数据，反之不成立。路由器具有 MAC 地址，能够读写数据帧，但其核心功能是 IP 数据包的路由转发，而非数据帧转发。

现代网络设备已突破传统层次界限。L4 交换机能够解析传输层端口号，实现负载均衡；L7 交换机工作在应用层，可执行更复杂的流量控制。

### 4、网关机制

网关是连接不同网络的出口。当主机发送数据包时，首先将目标 IP 地址与本机子网掩码进行比对：

- **局域网内通信**：目标 IP 在同一网段内，数据包通过交换机或 Wi-Fi 直接发送至目标主机，无需经过网关。
- **外部网络通信**：目标 IP 不在本地网段，主机无法直接到达。此时将数据包发送至默认网关，由网关处理后续转发。

比对结果影响 ARP 协议的行为。局域网内通信时，数据帧的目标 MAC 地址填写目标主机的 MAC 地址；跨网段通信时，目标 MAC 地址填写网关的 MAC 地址。

### 5、虚拟机网络模式

**桥接模式（Bridge）**：通过虚拟网桥（L2 设备）连接主机和虚拟机。虚拟机与主机处于同一局域网，使用相同网段的 IP 地址。

**NAT 模式**：通过虚拟路由器连接主机和虚拟机。虚拟机与主机处于不同局域网，使用不同网段的 IP 地址。虚拟路由器执行网络地址转换（NAT），使虚拟机能够访问外部网络。

## 二、TCP/IP 协议栈

### 1、协议族架构

TCP/IP 是协议族的统称，包含四个层次：网络接口层、网络层、传输层和应用层。每层提供不同的功能。

**IP 协议**：提供基本的命名方法和递送机制，能够在主机间传送数据报（Datagram）。IP 协议是不可靠的，数据报在网络中丢失或重复时不会尝试恢复。

**UDP 协议**：在 IP 协议基础上扩展，使数据包能够在进程间传送，而非仅在主机间传送。UDP 继承了 IP 的不可靠特性。

**TCP 协议**：构建在 IP 之上的复杂协议，提供进程间点对点的可靠双向连接。TCP 保证数据不丢失、不重复，按序到达。

### 2、套接字抽象

套接字（Socket）是连接的端点。从 Linux 程序角度看，套接字是具有文件描述符的打开文件，可通过标准 I/O 系统调用操作：`open`、`close`、`read`、`write`。

每个套接字具有对应的套接字地址，由 IP 地址和 16 位整数端口组成，表示为 `IP地址:端口`。

### 3、数据单元的层次划分

**传输层（TCP）**：数据单元称为 TCP 报文段（TCP Segment）。TCP 将应用层的长数据流切分为多个报文段。

**网络层（IP）**：数据单元称为 IP 数据包（IP Datagram/Packet）。当 IP 数据包超过链路的最大传输单元（MTU）时，执行分片操作。

## 三、TCP 协议原理

### 1、可靠传输的挑战

网络环境中存在多种问题：丢包、数据损坏、乱序、重复、链路拥塞、接收方拥塞，以及延迟和抖动。

TCP 的设计目标是在**不可靠**的网络环境中解决丢包、损坏、乱序和重复问题，建立**可靠连接**并提供**字节流**服务。

### 2、ARQ 机制与序列号

TCP 使用自动重传请求（ARQ）机制处理丢包和损坏问题。发送端需要确认两个状态：数据包是否丢失、数据包是否损坏。

**停止-等待协议**：发送方发送一个分组后等待 ACK。接收方收到分组后发送对应的 ACK。发送方收到 ACK 后发送下一个分组。超时未收到 ACK 则重传。

停止-等待协议存在的问题：
- 发送的分组可能丢失或损坏
- ACK 可能丢失或损坏
- 发送端无法区分分组丢失和 ACK 丢失

发送端对分组丢失和 ACK 丢失的处理方式相同：超时重传。接收端可能收到重复的分组（ACK 丢失或网络延迟导致）。

**序列号机制**：使用序列号解决重复问题。TCP 中的序列号表示每个分组的第一个字节在整个数据流中的字节偏移，本质是已发送字节的计数器。

停止-等待方式的吞吐量为 `分组大小 / 往返时间 = S / R`，效率低下。发送端、接收端和网络设备大部分时间处于空闲状态。

### 3、滑动窗口机制

提高吞吐量需要允许多个分组同时进入网络，采用流水线方式。使用滑动窗口机制后，吞吐量提升为 `窗口大小 × 分组大小 / 往返时间 = W × S / R`。

滑动窗口引入新的问题：除丢包和损坏外，还需处理乱序、接收方拥塞和网络设备拥塞。

**滑动窗口大小**：`swnd = min(rwnd, cwnd)`，其中 `rwnd` 是接收窗口（接收方通告），`cwnd` 是拥塞窗口（发送方维护）。

#### 滑动窗口的内核实现

Linux 内核中，滑动窗口通过指针在连续内存上划分区域实现。发送缓冲区被划分为四个动态区域：

1. **已发送、已确认（Category 1）**：窗口左侧外部的数据。这些数据已安全送达，可从内存释放。

2. **已发送、未确认（Category 2）**：窗口内的左侧部分。数据已发出，等待对方的 ACK。

3. **未发送、允许发送（Category 3）**：窗口内的右侧部分。接收方和网络有容量，发送方可立即发送这些数据。

4. **未发送、不允许发送（Category 4）**：窗口右侧外部的数据。超出当前 `swnd` 范围，由于流量或拥塞控制限制，不能发送。

**关键约束**：Category 2 + Category 3 的字节数之和严格等于 `swnd`。滑动窗口的范围由 `swnd` 决定。

#### 滑动窗口的滑动过程

假设当前 `swnd = min(rwnd, cwnd) = 1000` 字节，发送窗口大小为 1000 字节。

**步骤 1 - 发送数据**：发送方发送 Category 3 中的 400 字节数据。
- 这 400 字节从 Category 3 转变为 Category 2（已发送未确认）
- 窗口的左边界和右边界均未移动，仅内部数据状态改变

**步骤 2 - 收到 ACK**：接收方确认最早的 300 字节数据已收到，并在 TCP 头部的窗口字段通告新的接收窗口。计算后新的 `swnd` 仍为 1000 字节。

**步骤 3 - 窗口滑动**：
- 左边界右移：300 字节被确认，Category 1 的边界向右推进 300 字节
- 右边界右移：为维持窗口大小（`swnd = 1000` 字节），右边界同步向右推进 300 字节
- 结果：新腾出的 300 字节空间使原本属于 Category 4（不允许发送）的数据转变为 Category 3（允许发送）

**滑动机制总结**：
- 滑动窗口是机制框架
- `swnd` 是运行时的具体宽度限制
- 发送方任何时刻处于"已发送未确认"和"准备发送"状态的数据总量不能超过 `swnd`
- 当旧数据被确认，窗口按 `swnd` 指定的宽度向右滑动

### 4、连接管理

#### 三次握手

TCP 连接建立需要三次握手。三次握手足够让双方确认彼此的收发能力，四次握手无必要。

握手过程：
1. 客户端发送 SYN 报文
2. 服务器回复 SYN-ACK 报文
3. 客户端发送 ACK 报文

三次握手完成后，连接建立，双方可开始数据传输。

#### 四次挥手

TCP 连接终止需要四次挥手。一方发送 FIN 报文后，另一方可能仍有应用层数据需要发送。接收到 FIN 的一方先回复 ACK（"我知道了，但请等待"），待剩余数据发送完毕后再发送 FIN 报文。

挥手过程：
1. 主动关闭方发送 FIN 报文
2. 被动关闭方回复 ACK 报文
3. 被动关闭方发送 FIN 报文
4. 主动关闭方回复 ACK 报文

#### ACK 报文的重传机制

单独的 ACK 报文不占用序列号。第三次握手、第四次挥手或数据传输中的纯 ACK 报文不需要重传。只有 SYN、FIN 或携带数据的报文才占用序列号，需要重传。

单独 ACK 丢失的处理：依靠对方重传。对方未收到 ACK 时，定时器超时后重新发送原始报文。接收方收到重传报文后再次回复 ACK。

第四次挥手中，客户端发送最后一个纯 ACK 后需要等待 2MSL（Maximum Segment Lifetime）。这确保服务器能够收到 ACK，或在 ACK 丢失时能够接收并响应服务器的 FIN 重传。

## 四、Socket 编程接口

CSAPP 主要基于 TCP 讨论 Socket 编程。Socket 编程抽象了 TCP 的具体原理，提供标准的编程接口。

### 1、网络通信流程

**服务器端流程**：
1. 调用 `open_listenfd` 创建监听套接字，返回文件描述符 `listenfd`
2. 调用 `accept` 启动监听（阻塞调用），等待客户端连接
3. 客户端连接到达时，`accept` 返回已连接描述符 `connfd`
4. 通过 `connfd` 执行全双工读写操作
5. 调用 `close` 关闭连接，触发四次挥手

**客户端流程**：
1. 调用 `open_clientfd` 创建客户端套接字（内部执行 `connect`，阻塞调用）
2. `connect` 成功后，TCP 三次握手完成，连接建立
3. 通过返回的文件描述符执行全双工读写操作
4. 调用 `close` 关闭连接，触发四次挥手

### 2、连接的生命周期

1. **连接建立**：客户端调用 `connect`，服务器端 `accept` 返回，TCP 三次握手完成
2. **数据传输**：双方通过套接字描述符执行 `read` 和 `write` 操作
3. **连接终止**：任一方调用 `close`，TCP 四次挥手开始，连接断开

## 五、Web 服务器

### 1、前端与后端

Web 开发中的前端和后端分别对应客户端和服务器。前端代码可能存储在服务器上，但在客户端运行和渲染。

Web 应用相关的应用层协议包括 HTTP、WebSocket 和 FTP，其中 90% 以上的 Web 应用基于 HTTP 协议。

### 2、HTTP 协议

HTTP（超文本传输协议）基于 TCP 构建，用于传输各种内容，最常用的是 HTML 文件。

#### HTTP 与 HTML 的协作

浏览网页时，HTTP 和 HTML 的协作过程：

1. **发起请求（HTTP）**：浏览器作为客户端，按照 HTTP 规则向服务器发送请求，请求特定网页。

2. **返回响应（HTTP + HTML）**：服务器找到对应的 HTML 文件，将其封装在 HTTP 响应中发送回浏览器。

3. **渲染展示（HTML）**：浏览器解析 HTML 文件中的标签和代码，渲染为可视化网页。

HTML 负责构建网页内容，HTTP 负责传输网页内容。

#### HTTP 请求格式

HTTP 请求由三部分组成：请求行、请求头、空行（可选的请求体）。

**请求行格式**：`method URI version`

- `method`：请求方法，如 `GET`、`POST`、`OPTIONS` 等
- `URI`：统一资源标识符，指定请求的资源路径
- `version`：HTTP 协议版本，如 `HTTP/1.1`

**常用方法**：
- `GET`：读取资源
- `POST`：写入或提交数据

#### HTTP 响应格式

HTTP 响应由三部分组成：响应行、响应头、响应体。

**响应行格式**：`version status-code status-message`

- `version`：HTTP 协议版本
- `status-code`：状态码，如 `200`、`404`、`500` 等
- `status-message`：状态描述，如 `OK`、`Not Found`、`Internal Server Error` 等

### 3、静态内容服务

客户端请求静态内容时，服务器检索对应的文件并返回。

**请求示例**：`GET /index.html HTTP/1.1`

**处理流程**：
1. 服务器解析请求行，提取 URI `/index.html`
2. 在文件系统中定位 `index.html` 文件
3. 读取文件内容
4. 构造 HTTP 响应，将文件内容放入响应体
5. 通过已连接描述符发送响应给客户端

### 4、动态内容与 CGI

客户端请求动态内容时，服务器执行程序生成响应内容。

**请求示例**：`GET /cgi-bin/adder?15&17 HTTP/1.1`

**处理流程**：
1. 服务器解析请求行，识别 CGI 程序路径和查询参数
2. 服务器通过 `fork` 创建子进程
3. 子进程执行 `execve` 加载并运行 CGI 程序（如 `adder`）
4. 在 `fork` 之后、`execve` 之前，子进程设置环境变量传递信息给 CGI 程序（如 `QUERY_STRING=15&17`）
5. 子进程将标准输出重定向到已连接描述符 `connfd`（通过 `dup2`）
6. CGI 程序将响应输出到标准输出，自动发送给客户端

**CGI 标准**：定义了子进程与服务器进程之间的通信接口。CGI 程序通过环境变量接收请求信息，通过标准输出返回响应内容。


## 六、总结
网络编程建立在客户端-服务器模型之上，通过 TCP/IP 协议栈提供可靠的数据传输。TCP 协议使用滑动窗口、ARQ 机制和连接管理实现可靠传输。Socket 编程接口将网络通信抽象为文件操作，简化了网络应用开发。Web 服务器基于 HTTP 协议提供静态和动态内容服务，是网络编程的典型应用。
