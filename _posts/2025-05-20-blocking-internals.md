[同步/异步、阻塞/非阻塞：概念分析](./2025-05-20-sync-async-blocking-concepts.md)将阻塞定义为：调用者失去控制权，无法执行下一行代码。这是从调用者视角观察到的现象，称为**逻辑阻塞**。但同样表现为"控制流停在某行"，底层机制可能截然不同。本文分析造成逻辑阻塞的三种底层情形。

## 一、概念辨析

**逻辑阻塞**是现象描述：控制流停在某一行，无法推进到下一行。它不区分底层机制，是调用者的主观感受。

造成逻辑阻塞的底层机制分为两类：

- **内核睡眠**（也称内核级阻塞、阻塞）：线程因等待资源主动进入睡眠状态，`task_struct` 状态改变，让出 CPU，由内核调度唤醒。系统编程文档中提到的"阻塞"、"阻塞 I/O"通常特指此类。
- **用户态阻塞**：OS 线程状态保持 `TASK_RUNNING`，不涉及内核状态变化。又分为两种：
  - **cooperative suspension**（协程挂起）：让出用户态执行权，CPU 转去执行其他协程
  - **busy-waiting**（自旋等待）：占用 CPU 空转轮询

> "挂起"（suspend）在 Linux 中特指线程收到 `SIGSTOP` 或被 `ptrace` 跟踪后进入 `TASK_STOPPED` / `TASK_TRACED` 状态，是外部强制暂停，与内核睡眠不同。

## 二、内核睡眠

### 1、定义与特征

线程执行系统调用时，若所需条件不满足，线程主动进入睡眠，等待内核唤醒。

**特征**：

- 必须发生系统调用
- 线程的 `task_struct` 状态发生改变
- 线程让出物理 CPU 核心
- 内核调度其他线程运行

### 2、执行流程

线程执行系统调用（如 `read()`、`accept()`）时，若条件不满足（数据未到达），内核将线程从就绪队列移出，放入该事件的等待队列。线程状态从 `TASK_RUNNING` 变为 `TASK_INTERRUPTIBLE` 或 `TASK_UNINTERRUPTIBLE`。线程交出 CPU 控制权，直到事件发生被内核唤醒，重新进入就绪队列。

**示例**：调用 `read()` 读取套接字，若接收缓冲区无数据，线程进入等待队列，让出 CPU。网卡收到数据包后，内核将数据放入缓冲区，唤醒等待线程。

## 三、用户态阻塞

### 1、cooperative suspension：协程挂起

协程执行到 `co_await` 时，函数逻辑上停止推进，但物理线程未进入等待队列。协程框架将系统调用改为非阻塞模式，发现无数据时保存当前函数上下文，跳转执行同线程内的其他协程。OS 线程状态保持 `TASK_RUNNING`，内核对此无感知。

### 2、busy-waiting：自旋等待

线程尝试获取已占用的自旋锁时，在用户态执行 `while(lock == busy);` 循环。业务逻辑上函数未返回，但线程状态是 `TASK_RUNNING`，持续占用 CPU 执行等待循环。

## 四、三种情形对比

| 维度 | 内核睡眠 | cooperative suspension | busy-waiting |
|------|---------|----------------------|--------------|
| 常见叫法 | 阻塞、内核阻塞 | 协程挂起 | 自旋 |
| 线程状态 | 改变（`TASK_INTERRUPTIBLE` 等） | 不变（`TASK_RUNNING`） | 不变（`TASK_RUNNING`） |
| 是否让出 CPU | 是 | 否（转给其他协程） | 否（空转） |
| 是否发生系统调用 | 是 | 否 | 否 |
| 切换开销 | 大（内核态切换、页表刷新） | 小（函数上下文保存） | 无切换 |
| CPU 利用率 | 释放给其他线程 | 用于执行其他协程 | 浪费（空转） |

三者都会造成逻辑阻塞。区别在于 OS 线程是否让出 CPU，以及让出后 CPU 的去向。

## 五、I/O 操作的两个阶段

### 1、前置说明：用户态触发内核睡眠的唯一途径

在 Linux 的特权级隔离机制下，用户态代码（Ring 3）无权直接修改线程的 `task_struct` 状态，也无权调用内核调度器。用户态线程触发内核睡眠的唯一合法途径是**系统调用**。

常见触发场景：
- **I/O 阻塞**：`read`、`write`、`accept`、`connect`
- **多路复用阻塞**：`select`、`poll`、`epoll_wait`
- **同步原语阻塞**：`sem_wait`、`pthread_mutex_lock`（底层基于 `futex` 系统调用）
- **主动睡眠**：`nanosleep`、`clock_nanosleep`

本节重点分析 I/O 系统调用（`read` 和 `write`）的两阶段模型及其不对称性。

### 2、两阶段模型

I/O 过程涉及两个数据传输阶段，内核睡眠在此场景中有具体体现。

**硬件级 I/O**

数据在硬件设备与内核缓冲区之间传输。对于读操作，是设备→内核；对于写操作，是内核→设备。这一步涉及硬件 I/O，可能需要等待。

**内存级 Copy**

数据在内核缓冲区与用户进程内存之间拷贝。这是纯内存操作，由 CPU 执行（如 `copy_to_user`、`copy_from_user`）。

### 3、read 和 write 的不对称性

在 Linux 默认的同步阻塞 I/O 模式下，`read` 和 `write` 的行为是**不对称的**。这种不对称是操作系统为性能优化而设计的。

#### read：全链路同步

`read()` 的承诺：数据已从设备传输到内核缓冲区，并拷贝到用户内存。

**数据流向**：设备 → 内核缓冲区 → 用户内存（两步都是同步的）

**执行流程**：

1. **硬件级 I/O（设备→内核）**：若内核缓冲区无数据，线程进入等待队列，发生**内核睡眠**。等待硬件（网卡、磁盘）将数据传输到内核缓冲区。
2. **内存级 Copy（内核→用户）**：数据就绪后，内核调用 `copy_to_user()` 将数据从内核缓冲区拷贝到用户空间。这是纯内存操作，**不会发生内核睡眠**，但会占用少量 CPU 时间。

`read()` 返回时，数据已完整地位于用户内存中，可立即使用。

#### write：部分同步

`write()` 的承诺：数据已从用户内存拷贝到内核缓冲区。至于数据何时从内核缓冲区传输到设备，由内核异步完成，`write()` 不保证。

**数据流向**：用户内存 → 内核缓冲区（同步）→ 设备（异步）

**执行流程**：

1. **内存级 Copy（用户→内核）**：内核调用 `copy_from_user()` 将数据从用户空间拷贝到内核缓冲区。若内核缓冲区已满，线程进入等待队列，发生**内核睡眠**。
2. **硬件级 I/O（内核→设备）**：内核后台异步将数据刷新到设备（如磁盘的脏页回写、网卡的数据包发送）。`write()` 返回时，这一步可能尚未完成。

`write()` 返回时，数据已在内核缓冲区中，但可能尚未到达硬件设备。若此时断电，数据可能丢失。

#### 不对称的原因

- **写操作追求速度**：用户程序通常不需要等待数据真正落盘，只需确保数据已交给内核管理。内核用廉价的内存缓冲区掩盖昂贵的硬件 I/O 延迟。
- **读操作追求完整性**：用户程序必须拿到真实数据才能进行后续计算，内核无法凭空变出数据，必须等待硬件传输完成。

**特例**：使用 `O_SYNC` 标志或调用 `fsync()` 可强制 `write()` 等待数据落盘，实现与 `read()` 对称的全链路同步。

### 4、同步 I/O 与异步 I/O 的判断标准

根据 POSIX 定义，判断标准是：**该 I/O 操作是否会阻塞当前线程，直到系统调用承诺的任务完成**。

**同步 I/O**：线程必须等待系统调用承诺的任务完成才能继续执行。线程需要等待内存级 Copy 完成。

- `read()`：等待硬件级 I/O（设备→内核）和内存级 Copy（内核→用户）全部完成
- `write()`：等待内存级 Copy（用户→内核）完成（硬件级 I/O 是异步的，但不影响 `write()` 本身是同步 I/O）
- `select()`、`epoll_wait()`：等待 fd 就绪，返回后仍需调用 `read()` 完成内存级 Copy

**异步 I/O**：线程发起 I/O 请求后立即返回，内核完成硬件级 I/O 和内存级 Copy 的全部工作后，通过信号或队列通知线程。线程全程无需等待。

- Linux 的 `io_uring`、Windows 的 IOCP 属于此类

### 5、多路复用中的 fd 就绪状态

在 I/O 多路复用（`select`、`poll`、`epoll`）中，"fd 就绪"（Readable/Writable）的含义与前述两个传输环节直接相关。理解这一点有助于正确使用多路复用机制。

#### Readable：可读就绪

**含义**：内核缓冲区有数据可读。

当多路复用返回某个 fd 为 Readable 时，表示：
- **硬件级 I/O 已完成**：数据已从设备传输到内核缓冲区
- **调用 `read()` 不会阻塞**：`read()` 可以立即执行内存级 Copy，将数据从内核缓冲区拷贝到用户空间，无需等待硬件

**典型场景**：
- 套接字：网卡已将数据包放入接收缓冲区
- 管道：写端已写入数据到管道缓冲区
- 普通文件：数据已在 Page Cache 中（普通文件总是就绪的）

#### Writable：可写就绪

**含义**：内核缓冲区有空间可写。

当多路复用返回某个 fd 为 Writable 时，表示：
- **内核缓冲区未满**：有足够空间接收用户数据
- **调用 `write()` 不会阻塞**：`write()` 可以立即执行内存级 Copy，将数据从用户空间拷贝到内核缓冲区，无需等待缓冲区腾出空间

**典型场景**：
- 套接字：发送缓冲区有空间（对端接收正常，网络未拥堵）
- 管道：管道缓冲区未满（读端正在读取）
- 普通文件：Page Cache 有空间（普通文件通常总是可写的）

#### 关键点

1. **fd 就绪不等于硬件就绪**：Readable 只保证内核缓冲区有数据，不保证数据已到达最终目的地；Writable 只保证内核缓冲区有空间，不保证数据已发送到设备。

2. **内存级 Copy 仍需等待**：即使 fd 就绪，调用 `read()`/`write()` 后，线程仍需等待内核完成内存级 Copy（用户空间与内核空间之间的数据拷贝）。这就是为什么多路复用属于同步 I/O。

4. **与 read/write 不对称性的关系**：
   - `read()` 的 Readable：保证硬件级 I/O 已完成，内存级 Copy 可立即执行
   - `write()` 的 Writable：保证内存级 Copy 可立即执行，硬件级 I/O 由内核异步完成

**示例**：

```c
// 使用 epoll 监控套接字
struct epoll_event ev;
epoll_wait(epfd, &ev, 1, -1);  // 阻塞等待 fd 就绪

if (ev.events & EPOLLIN) {
    // fd 可读：内核缓冲区有数据
    read(fd, buf, size);  // 不会因等待硬件而阻塞，但会等待内核拷贝数据
}

if (ev.events & EPOLLOUT) {
    // fd 可写：内核缓冲区有空间
    write(fd, buf, size);  // 不会因缓冲区满而阻塞，但会等待内核接收数据
}
```

## 六、select 机制：内核睡眠的典型案例

`select()` 是 I/O 多路复用的经典系统调用，允许单个线程同时监控多个文件描述符。其内部实现完整体现了内核睡眠的流程。

### 1、工作流程

**步骤 1：拷贝 fd 集合**

将用户空间的 fd 位图拷贝到内核空间。

**步骤 2：第一次轮询**

内核遍历每个 fd，调用对应驱动的 `poll` 函数，检查缓冲区状态。若有 fd 已就绪，在返回位图中标记并返回。

**步骤 3：挂载等待队列**

若无 fd 就绪，内核将当前线程挂载到每个 fd 的等待队列。任意 fd 就绪时均可唤醒该线程。

**步骤 4：线程睡眠**

内核将线程状态改为 `TASK_INTERRUPTIBLE`，让出 CPU。此处发生内核睡眠（即阻塞）。

**步骤 5：唤醒与返回**

某个 fd 就绪（如网卡数据到达），内核唤醒睡眠线程。线程再次遍历 fd 集合，标记就绪的 fd，返回用户态。

### 2、select 属于同步 I/O

`select` 返回后，线程必须亲自调用 `read()`，等待数据从内核空间拷贝到用户空间。阶段 2 期间线程原地等待，符合同步 I/O 的定义。

真正的异步 I/O（如 `io_uring`）中，线程调用后直接离开，内核完成数据等待和拷贝的全部工作，完成后主动通知线程。

### 3、select 的性能问题

- **重复拷贝**：每次调用都将整个 fd 集合从用户态拷贝到内核态
- **盲目轮询**：线程唤醒后不知道哪个 fd 就绪，需 O(N) 遍历所有 fd
- **重复挂载**：每次调用都将线程挂载到所有 fd 的等待队列，返回时拆除

`epoll` 通过维护持久的红黑树和就绪链表解决了上述问题。fd 就绪时内核直接将其放入就绪链表，线程醒来直接从链表获取，无需轮询。
