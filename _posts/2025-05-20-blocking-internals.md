[同步/异步、阻塞/非阻塞：概念分析](./2025-05-20-sync-async-blocking-concepts.md)将阻塞定义为：调用者失去控制权，无法执行下一行代码。这是从调用者视角观察到的现象，称为**逻辑阻塞**。但同样表现为"控制流停在某行"，底层机制可能截然不同。本文分析造成逻辑阻塞的三种底层情形。

## 一、概念辨析

**逻辑阻塞**是现象描述：控制流停在某一行，无法推进到下一行。它不区分底层机制，是调用者的主观感受。

造成逻辑阻塞的底层机制分为两类：

- **内核睡眠**（也称内核阻塞、阻塞）：线程因等待资源主动进入睡眠状态，`task_struct` 状态改变，让出 CPU，由内核调度唤醒。系统编程文档中提到的"阻塞"、"阻塞 I/O"通常特指此类。
- **用户态阻塞**：OS 线程状态保持 `TASK_RUNNING`，不涉及内核状态变化。又分为两种：
  - **cooperative suspension**（协程挂起）：让出用户态执行权，CPU 转去执行其他协程
  - **busy-waiting**（自旋等待）：占用 CPU 空转轮询

> "挂起"（suspend）在 Linux 中特指线程收到 `SIGSTOP` 或被 `ptrace` 跟踪后进入 `TASK_STOPPED` / `TASK_TRACED` 状态，是外部强制暂停，与内核睡眠不同。

## 二、内核睡眠

### 1、定义与特征

线程执行系统调用时，若所需条件不满足，线程主动进入睡眠，等待内核唤醒。

**特征**：

- 必须发生系统调用
- 线程的 `task_struct` 状态发生改变
- 线程让出物理 CPU 核心
- 内核调度其他线程运行

### 2、执行流程

线程执行系统调用（如 `read()`、`accept()`）时，若条件不满足（数据未到达），内核将线程从就绪队列移出，放入该事件的等待队列。线程状态从 `TASK_RUNNING` 变为 `TASK_INTERRUPTIBLE` 或 `TASK_UNINTERRUPTIBLE`。线程交出 CPU 控制权，直到事件发生被内核唤醒，重新进入就绪队列。

**示例**：调用 `read()` 读取套接字，若接收缓冲区无数据，线程进入等待队列，让出 CPU。网卡收到数据包后，内核将数据放入缓冲区，唤醒等待线程。

## 三、用户态阻塞

### 1、cooperative suspension：协程挂起

协程执行到 `co_await` 时，函数逻辑上停止推进，但物理线程未进入等待队列。协程框架将系统调用改为非阻塞模式，发现无数据时保存当前函数上下文，跳转执行同线程内的其他协程。OS 线程状态保持 `TASK_RUNNING`，内核对此无感知。

### 2、busy-waiting：自旋等待

线程尝试获取已占用的自旋锁时，在用户态执行 `while(lock == busy);` 循环。业务逻辑上函数未返回，但线程状态是 `TASK_RUNNING`，持续占用 CPU 执行等待循环。

## 四、三种情形对比

| 维度 | 内核睡眠 | cooperative suspension | busy-waiting |
|------|---------|----------------------|--------------|
| 常见叫法 | 阻塞、内核阻塞 | 协程挂起 | 自旋 |
| 线程状态 | 改变（`TASK_INTERRUPTIBLE` 等） | 不变（`TASK_RUNNING`） | 不变（`TASK_RUNNING`） |
| 是否让出 CPU | 是 | 否（转给其他协程） | 否（空转） |
| 是否发生系统调用 | 是 | 否 | 否 |
| 切换开销 | 大（内核态切换、页表刷新） | 小（函数上下文保存） | 无切换 |
| CPU 利用率 | 释放给其他线程 | 用于执行其他协程 | 浪费（空转） |

三者都会造成逻辑阻塞。区别在于 OS 线程是否让出 CPU，以及让出后 CPU 的去向。

## 五、I/O 操作的两个阶段

I/O 过程分为两个阶段，内核睡眠在此场景中有具体体现。

**阶段 1：等待数据准备**

网卡收到数据包并复制到内核缓冲区，或磁盘控制器将数据读入内核缓冲区。

**阶段 2：数据拷贝**

将数据从内核缓冲区复制到用户空间的进程内存。

**同步 I/O 的判断标准**：阶段 2 期间，进程需要停下来等待内核完成数据拷贝，即为同步 I/O。进程参与了拷贝过程的等待。

**异步 I/O 的判断标准**：内核将阶段 1 和阶段 2 全部完成，直接将数据放入用户空间并通知进程。进程全程无需等待拷贝。Linux 的 `io_uring` 和 Windows 的 IOCP 属于此类。

## 六、select 机制：内核睡眠的典型案例

`select()` 是 I/O 多路复用的经典系统调用，允许单个线程同时监控多个文件描述符。其内部实现完整体现了内核睡眠的流程。

### 1、工作流程

**步骤 1：拷贝 fd 集合**

将用户空间的 fd 位图拷贝到内核空间。

**步骤 2：第一次轮询**

内核遍历每个 fd，调用对应驱动的 `poll` 函数，检查缓冲区状态。若有 fd 已就绪，在返回位图中标记并返回。

**步骤 3：挂载等待队列**

若无 fd 就绪，内核将当前线程挂载到每个 fd 的等待队列。任意 fd 就绪时均可唤醒该线程。

**步骤 4：线程睡眠**

内核将线程状态改为 `TASK_INTERRUPTIBLE`，让出 CPU。此处发生内核睡眠（即阻塞）。

**步骤 5：唤醒与返回**

某个 fd 就绪（如网卡数据到达），内核唤醒睡眠线程。线程再次遍历 fd 集合，标记就绪的 fd，返回用户态。

### 2、select 属于同步 I/O

`select` 返回后，线程必须亲自调用 `read()`，等待数据从内核空间拷贝到用户空间。阶段 2 期间线程原地等待，符合同步 I/O 的定义。

真正的异步 I/O（如 `io_uring`）中，线程调用后直接离开，内核完成数据等待和拷贝的全部工作，完成后主动通知线程。

### 3、select 的性能问题

- **重复拷贝**：每次调用都将整个 fd 集合从用户态拷贝到内核态
- **盲目轮询**：线程唤醒后不知道哪个 fd 就绪，需 O(N) 遍历所有 fd
- **重复挂载**：每次调用都将线程挂载到所有 fd 的等待队列，返回时拆除

`epoll` 通过维护持久的红黑树和就绪链表解决了上述问题。fd 就绪时内核直接将其放入就绪链表，线程醒来直接从链表获取，无需轮询。
