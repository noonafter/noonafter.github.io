# 博客模糊搜索路线图

## 当前基础

第一阶段在 Jekyll 构建时生成文章标题、URL 和 Markdown 正文索引，在浏览器中完成不区分大小写的逐行字面搜索。整个功能保持纯静态，不依赖服务器或外部搜索服务。

## 第二阶段目标

在保留精确文本搜索的基础上增加模糊匹配，主要处理英文拼写错误、少量中文错字、漏字以及不完整关键词，并按相关性排序结果。

## 推荐方案

1. 继续复用当前构建期生成的正文索引，不改变 GitHub Pages 部署方式。
2. 将文章拆分为标题和可搜索文本行，以文本行为最小匹配单元。
3. 优先使用体积较小、支持浏览器端运行的模糊搜索实现，并将依赖文件存放在站点本地，避免 CDN 可用性影响搜索。
4. 排序权重建议为：标题精确匹配 > 标题模糊匹配 > 正文精确匹配 > 正文模糊匹配。
5. 对短查询设置限制，例如少于两个字符时仅执行精确搜索，避免中文单字产生大量低质量结果。
6. 将索引初始化放在搜索脚本按需加载之后，避免影响普通页面的首屏加载。

## 候选实现

- 首选：MiniSearch。适合字段权重、前缀搜索、模糊参数和结果排序。
- 备选：Fuse.js。集成简单，适合当前数据量，但需要额外控制全文逐行搜索的结果数量与性能。
- 不建议自行实现完整编辑距离索引；维护成本高，中文效果也不一定优于成熟库。

最终选型前应使用博客的真实文章索引做基准测试，比较索引脚本体积、初始化耗时、典型查询耗时和结果质量。

## 中文搜索说明

普通模糊搜索只能处理原始汉字的近似匹配，不等同于拼音搜索。例如输入 `neicun` 无法自然匹配“内存”。如果需要拼音搜索，应作为独立增强：构建时为标题和关键词生成拼音字段，并同时保留原文索引。正文全文生成拼音会显著增大索引，初期建议只对标题、标签和文章关键词启用。

## 验收标准

- 精确搜索结果不退化，仍能显示命中行和高亮。
- 常见英文错拼、中文漏字能够找到预期文章。
- 82 篇文章规模下，桌面端和移动端输入搜索时没有明显卡顿。
- 模糊结果具有稳定、可解释的排序，并限制每篇文章和全局结果数量。
- 不依赖后端、数据库和外部在线服务。
